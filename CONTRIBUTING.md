# Участие в разработке

Спасибо за интерес к net-test! Несколько правил, чтобы изменения проходили гладко.

## Требования

- Go **1.26+** (минимум из `go.mod`).
- `make` (опционально, но удобно).
- Зависимости тянутся автоматически (`go.mod`); ничего ставить вручную не нужно.

## Локальный цикл

```bash
make build         # собрать ./net-test
make run           # запустить (ARGS="-target 8.8.8.8")
make test          # юнит-тесты (без сети)
make test-race     # тесты с детектором гонок
make live          # живой сетевой тест: пинг + трасса
make fmt           # gofmt -w .
make vet           # go vet ./...
```

Перед PR прогоните как минимум `make fmt vet test`, а при правках в
`internal/probe` — ещё и `make test-race` и `make live`; при правках в
`mobile/app` — `make test-mobile`.

## Стиль кода

- **Форматирование:** только `gofmt` (`make fmt`). PR с неотформатированным кодом
  не принимаются.
- **Имена и комментарии:** как в окружающем коде. Экспортируемые типы и функции
  снабжаются doc-комментариями; комментарии объясняют «почему», а не «что».
- **Зависимости:** новые сторонние модули — только при явной необходимости и с
  обоснованием в PR. Цель — лёгкий, почти-stdlib набор.
- **Платформенный код** живёт за интерфейсом `prober` под build-тегами. Не
  добавляйте `runtime.GOOS`-ветвления в общий код — см.
  [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md).

## Слои

- `internal/probe` не импортирует `internal/ui` и не знает про терминал. Здесь
  живут и измерения, и аналитика (аномалии, диагноз, ASN) — наружу отдаются
  готовые поля в снапшотах через каналы.
- `internal/ui` — только отображение готовых снапшотов.
- `internal/report` — отдельный wire-формат для `--once`/`--json`; меняйте его
  JSON-теги осознанно, это публичный контракт.
- `mobile/app` — Android/desktop GUI на Fyne, **отдельный модуль** (свой `go.mod`,
  тяжёлые зависимости Fyne не текут в ядро). Импортирует `internal/probe` напрямую;
  сборка `make apk`/`make gui` (нужны CGO + Android NDK), тесты — `make test-mobile`.

См. [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md).

## Коммиты и PR

- Коммиты — небольшие и осмысленные; сообщение в повелительном наклонении
  («add windows backend», а не «added»).
- Один PR — одно логическое изменение. Опишите, что и зачем, и как проверяли
  (особенно если меняли сетевой слой и тестировали на конкретной ОС).
- Если правите платформенный бэкенд — укажите, на какой ОС реально гоняли рантайм
  (компиляции под `GOOS` недостаточно).

## Сообщить о проблеме

Заводите [issue](https://github.com/tavvet/net-test/issues) с:

- ОС и архитектурой (`uname -a` / версия Windows), версией (`net-test -version`);
- командой запуска и целью (`-target`);
- что ожидали и что получили (текст из поля ошибки во вкладке, если есть).
